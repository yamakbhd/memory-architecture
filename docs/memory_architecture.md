# 记忆架构（Memory Architecture）

## 概述

记忆架构是 AI Agent 的核心基础设施，负责在不同粒度和时间跨度上管理和组织信息。本架构将记忆划分为五个层次，从长周期的项目记忆到即时的事件记忆，形成一个自顶向下的信息流通道，同时在每一层通过有损压缩和上下文窗口管理来控制在通用模型中的 token 消耗。

架构设计的核心原则：

- **分层递进**：记忆从宏观（项目级）到微观（操作级）逐层细化，上层为下层提供上下文约束，下层为上层提供执行反馈
- **有损压缩**：长期/海量信息经过摘要和结构化后才能进入有限上下文窗口
- **可检索性**：长期记忆支持语义检索和结构化查询
- **跨会话持久化**：项目级和部分对话级记忆在会话结束后持久化，支持跨 Session 复用

---

## 架构总览

整个记忆架构自上而下分为五大层：

```
+-------------------------------------------------------------------+
|                   1. 项目记忆 (Project Memory)                      |

+-------------------------------------------------------------------+
|                   2. 规划记忆 (Plan Memory)                         |

| 依赖检测和动态调度 · 优先级冲突解决                                      |
+-------------------------------------------------------------------+
|                   3. 对话记忆 (Session Memory)                      |
| +---------------------------------------------------------------+ |
| |              长期记忆 (Long-term Memory)                         | |
| |  · 知识库 · 经验库 · 行为库                                        | |
| | +------------------------------------------------------------+ | |

| | | +--------------------------------------------------------+ | | |
| | | |        短期记忆--有损 (Limited STM)                       | | | |

| | | |  · 上轮代码变更保留                                        | | | |
| | | | +-----------------------------------------------------+ | | | |
| | | | |       当前任务记忆 (Task Memory)                       | | | | |
| | | | |  · 代码片段 · 最近行为和结果 · 计划步骤序列                | | | | |
| | | | +-----------------------------------------------------+ | | | |

| 事件循环序列 · 交互日志 · 任务状态变化 · 工具调用注册                      |
+-------------------------------------------------------------------+
|                   5. 训练接口 (Training Interface)                    |
| 在线学习管线 · 离线调优基准                                             |
+-------------------------------------------------------------------+
```

## 1. 项目记忆（Project Memory）

项目记忆是最顶层的持久化记忆层，负责管理多个独立项目的全生命周期上下文。

### 1.1 核心特性

- **多项目独立管理，项目间隔离**：每个项目拥有独立的记忆命名空间，项目 A 的对话上下文不会被项目 B 污染。项目通过唯一的 project_id 标识和隔离。
- **跨项目共享记忆**：在隔离的基础上，允许通过显式的共享声明将某些通用经验（如通用工具使用模式、通用编码规范）标记为跨项目可复用，供其他项目检索。
- **时间线和优先级管理**：每个项目维护独立的进度时间线和优先级标记，支持在多项目间进行资源调度。
- **支持并行任务执行**：多个项目可以同时处于活跃状态，Agent 在切换项目时可以快速恢复对应项目的上下文。

### 1.2 数据结构

```json
ProjectMemory {
  project_id: string,
  name: string,
  status: "active" | "paused" | "completed",
  created_at: timestamp,
  updated_at: timestamp,
  priority: int (1-10),
  context_summary: string (项目核心上下文摘要，约 500-2000 tokens),
  shared_knowledge_keys: string[] (跨项目共享的知识条目标识),
  file_tree_snapshot: json (项目文件结构快照),
  timeline: TimelineEntry[]
}
```

### 1.3 运作机制

1. **项目创建**：Agent 开始新任务时，在项目记忆中创建项目记录，初始化上下文摘要
2. **项目切换**：Agent 在不同项目间切换时，从项目记忆中加载目标项目的 context_summary，注入到当前对话上下文中
3. **上下文更新**：每个 Planning-Session 结束时，将 context_summary 更新为最新状态
4. **跨项目检索**：当 Agent 遇到通用型问题（如特定工具的使用方式），可以在 shared_knowledge_keys 中检索跨项目共享的经验

## 2. 规划记忆（Plan Memory）

规划记忆位于项目记忆之下，负责将项目级的宏观目标分解为可执行的微观任务。

### 2.1 核心特性

- **多轮任务规划和执行**：任务不是一次性规划完成的，而是采用 规划 -> 执行 -> 反馈 -> 再规划 的迭代循环，每一轮规划的结果都存储在规划记忆中。
- **自动分层分解**：将宏观任务自动分解为子任务、步骤和子步骤的层次结构。分解遵循单一职责原则：一个叶子节点应该是一个可被单独完成的原子任务。
- **依赖检测和动态调度**：识别子任务间的依赖关系（如文件 B 的修改依赖文件 A 的重构），在满足依赖条件时动态调度可执行的子任务。
- **优先级冲突解决**：当多个子任务同时可执行但资源有限时，根据优先级规则（用户指定 > 核心功能 > 边缘功能 > 优化）确定执行顺序。

### 2.2 数据结构

```json
PlanMemory {
  plan_id: string,
  project_id: string,
  round: int (当前轮次),
  status: "planning" | "executing" | "completed" | "blocked",
  task_tree: TaskNode {
    description: string,
    subtasks: TaskNode[],
    progress: 0.0 - 1.0
  },
  dependency_graph: {
    nodes: DepNode[],
    edges: DepEdge[]
  },
  priority_queue: PrioritizedTask[],
  plan_history: HistoricalPlan[] (保留前 N 轮的规划快照)
}
```

### 2.3 运作机制

1. **初始规划**：Agent 接收用户请求后，在规划记忆中生成初始任务树
2. **执行与反馈**：每完成一个子任务，更新 progress 和 status；如果执行过程中发现新的子任务需求，插入任务树
3. **异常重规划**：如果子任务执行失败或被阻塞，触发局部重规划，只调整受影响的分支
4. **上下文注入**：当前活跃的子任务及其依赖关系注入到对话记忆的短期记忆中

## 3. 对话记忆（Session Memory）

对话记忆是记忆架构的核心层，管理 AI 模型在单次或多次对话中的所有信息。它由四个同心子层组成，从内到外粒度逐渐扩大、信息密度逐渐降低。

### 3.1 长期记忆（Long-term Memory / LTM）

长期记忆是跨会话持久化的知识积累层，存储从过去对话中提取的可复用知识、经验和行为模式。

#### 3.1.1 知识库（Knowledge Base）

- **NLP 知识**：自然语言处理的领域知识，如代码解析规则、语法模式、语义理解框架
- **语义网**：实体间的语义关系图谱，如项目文件间的引用依赖关系、API 调用关系
- **神经符号集成**：将神经网络的模式识别结果与符号化规则结合形成可理解的知识表达
- **交互式优化**：通过与用户的持续交互，逐步校正和完善知识条目（如用户纠正了某个工具的使用方式）

#### 3.1.2 经验库（Experience Base）

- **分解策略**：记录针对不同类型任务的有效分解方法和历史成功案例
- **性能/错误模式**：记录常见性能瓶颈和编码错误模式及其解决方案
- **偏好模型**：记录用户/团队的编码风格偏好、命名习惯、架构选择倾向

#### 3.1.3 行为库（Behavior Base）

- **行动模式**：抽象的可复用操作序列（如 查找->分析->修改->测试->验证 五步法）
- **工作流模板**：针对特定场景的标准化工作流（如代码审查流程、发布检查清单）
- **宏执行**：可被条件触发的自动化操作序列（如检测到特定文件变更时自动运行相关测试）

#### 3.1.4 数据结构

```json
LongTermMemory {
  knowledge_base: {
    entries: KnowledgeEntry[]  // {id, topic, content, embedding, source_conversation_id, verified_at}
  },
  experience_base: {
    entries: ExperienceEntry[] // {id, type, pattern, solution, success_rate, usage_count}
  },
  behavior_base: {
    patterns: BehaviorPattern[] // {id, trigger_condition, action_sequence, context_requirement}
    workflows: Workflow[]       // {id, name, steps, precondition, postcondition}
    macros: Macro[]             // {id, trigger, action_list}
  }
}
```

### 3.2 短期记忆（Short-term Memory / STM）

短期记忆管理当前对话会话中的完整上下文，是 LLM 上下文窗口的主要数据来源。

- **完整对话历史**：当前 Session 中所有用户消息和 Agent 响应的完整记录
- **用户信息**：当前用户的基本信息、权限级别、历史交互摘要
- **工具信息**：当前可用工具集、每个工具的参数签名和使用说明

### 3.3 短期记忆——有损（Limited STM）

当对话长度超过模型上下文窗口限制时，有损短期记忆通过摘要和选择性保留来压缩信息。

- **上轮规划阶段对话摘要**：将上一轮 Planning 阶段的完整对话压缩为结构化摘要（约 200-500 tokens），保留关键决策、变更意图和异常信息
- **上轮代码变更保留**：保留上一轮执行阶段中的代码 diff（有选择地保留关键变更，丢弃中间探索性修改）

### 3.4 当前任务记忆（Current Task Memory）

当前任务记忆是对话记忆的最内层，直接注入 LLM 的上下文窗口，包含执行当前子任务所需的最小信息集。

- **代码片段**：当前任务涉及的代码文件内容或关键代码段
- **最近行为和结果**：Agent 最近执行的操作及其结果（如工具调用的输入输出）
- **计划步骤序列**：当前子任务的步骤列表，标记已完成、进行中和待执行的步骤

### 3.5 各子层对比

| 层级 | 范围 | 持久化 | Token 量 | 更新频率 |
|------|------|--------|----------|----------|
| LTM 长期记忆 | 跨 Session | 是（向量数据库/知识图谱） | 不直接注入，通过检索注入 | 低频（Session 结束时更新） |
| STM 短期记忆 | 当前 Session | 否（Session 内有效） | 10K-30K tokens | 实时追加 |
| Limited STM | 上一轮 | 临时（压缩后废弃原始） | 200-500 tokens | 每轮 Planning 结束时压缩 |
| Task Memory | 当前任务 | 否（任务内有效） | 1K-5K tokens | 任务开始/结束时更新 |

## 4. 事件记忆（Event Memory）

事件记忆是架构的底层记录层，以事件为单位记录 Agent 在运行过程中的所有可观测行为，是复盘、审计和反馈学习的数据基础。

### 4.1 核心特性

- **事件循环序列**：以 Event Loop 为周期记录 Agent 的 "感知 -> 推理 -> 行动 -> 观察" 完整循环
- **交互日志**：记录 Agent 与用户、工具、外部系统的每一次交互
- **任务状态变化**：记录所有任务/子任务的状态转换（pending -> in_progress -> completed / failed）
- **工具调用注册**：记录每一次工具调用的函数名、参数、返回值、执行耗时

### 4.2 数据结构

```json
Event {
  event_id: string,
  timestamp: datetime,
  type: "user_message" | "agent_response" | "tool_call" | "tool_result" | "state_change" | "error" | "plan_update",
  payload: json,
  session_id: string,
  task_context: { plan_id: string, task_id: string }
}

EventLoop {
  loop_id: string,
  session_id: string,
  start_time: datetime,
  end_time: datetime,
  events: Event[] (该循环内所有事件),
  summary: string (循环摘要)
}
```

### 4.3 与训练接口的关系

事件记忆直接对接训练接口，将原始事件流经过清洗和标注后输入在线学习管线，同时导出到离线调优基准数据集中。

## 5. 训练接口（Training Interface）

训练接口是记忆架构与模型训练/调优之间的桥梁，负责将记忆数据转化为可训练的样本。

### 5.1 在线学习管线

- **实时反馈采集**：从事件记忆中采集 Agent 的执行成败信号和用户反馈信号（如用户直接纠正、用户接受结果）
- **样本生成**：将 "上下文 + 行动 + 结果" 三元组转化为强化学习或偏好学习样本
- **权重更新**：将新样本注入模型训练循环，逐步优化 Agent 行为

### 5.2 离线调优基准

- **数据集构建**：从事件记忆中批量导出历史交互记录，经人工或自动标注后构建评测基准
- **AB 测试框架**：保留不同版本 Agent 在相同任务上的事件流，用于对比分析
- **退化检测**：通过基准测试自动化检测 Agent 在新版本上的退化

## 6. 数据流与交互

### 6.1 自顶向下流（Top-Down）

1. **项目记忆** 提供项目的全局上下文 -> 注入到 **规划记忆** 作为任务分解的约束条件
2. **规划记忆** 提供当前活跃任务树和子任务 -> 注入到 **对话记忆** 的短期和当前任务记忆中
3. **对话记忆** 提供完整的执行上下文 -> LLM 在此上下文中生成具体的操作指令

### 6.2 自底向上流（Bottom-Up）

1. **事件记忆** 记录每一次单步操作的完整事件数据
2. 每个 Planning Cycle 结束时，从事件记忆中提取本轮的关键事件 -> 更新 **对话记忆** 的长期记忆
3. 每个 Session 结束时，从对话记忆中提取新的经验/知识 -> 更新 **项目记忆**
4. 从持久化记忆中导出数据 -> 输入 **训练接口** 进行模型优化

### 6.3 关键交互协议

```
// Agent 启动 Session
ProjectMemory -> load_context() -> inject_to_session()
  PlanMemory -> resume_or_create_plan() -> inject_current_tasks()
    SessionMemory.STM <- load_full_history()
    SessionMemory.TaskMemory <- load_current_task()

// Agent 执行单步操作
Agent.act() -> EventMemory.record(event)
  - 如果任务完成 -> PlanMemory.update_progress()
  - 如果任务失败 -> PlanMemory.replan()

// Session 结束
SessionMemory -> summarize_and_compress()
  -> LongTermMemory.update()
  -> ProjectMemory.update_context_summary()
  -> TrainingInterface.export_samples()
```

## 7. 实现要点

### 7.1 存储选型

| 记忆层 | 推荐存储 | 理由 |
|--------|----------|------|
| 项目记忆 | SQLite / PostgreSQL | 结构化数据，需要事务支持和关系查询 |
| 规划记忆 | SQLite + JSON | 树形结构以 JSON 存储，元数据用关系表 |
| 长期记忆 | 向量数据库（Chroma / Pinecone） | 语义检索需求，需支持 embedding 相似度搜索 |
| 短期/任务记忆 | 内存（Redis 可选） | 高频读写，不需要持久化（Session 级生命周期） |
| 事件记忆 | PostgreSQL / ClickHouse | 高频写入的事件流，需要时间序列查询能力 |

### 7.2 上下文窗口管理

- **Token 预算机制**：为每个记忆层设定最大 token 配额，超出配额时触发压缩
- **优先级驱逐**：当 token 预算不足时，按优先级逐出：Task Memory > Limited STM > STM > LTM 检索结果 > Plan Memory > Project Memory
- **滚动窗口**：短期记忆保留最近 N 轮对话的完整历史，更早的轮次自动压缩到 Limited STM

### 7.3 检索策略

- **长期记忆检索**：在每次 Planning 开始时，基于当前任务描述进行语义检索，返回 Top-K 相关条目
- **经验库检索**：匹配当前任务的类型标签和描述关键词，返回相似历史案例
- **行为库触发**：事件循环中检测到条件匹配时自动触发对应宏或工作流
